# 🖥️ 입출력 다중화 (I/O Multiplexing) 최종 요약

### 1. 핵심 개념
> **"CPU 타임 외의 작업들을 커널에게 맡기고, 우리는 커널이 알려줄 때만 작업하자!"**

* **기본 원리**: 프로세스가 수많은 파일(소켓)을 직접 하나하나 확인하며 기다리지 않습니다.
* **이벤트 기반**: 데이터가 도착하거나 연결이 완료되는 등 **'CPU 연산 외의 상태 변화'**가 일어났을 때 커널이 신호를 주면 그제야 CPU가 움직입니다.



---

### 2. 용도 및 실제 사례
* **대규모 네트워크 서비스**: 한정된 자원으로 수만 명의 동시 접속자를 처리할 때 (Nginx).
* **고성능 시스템 소프트웨어**: **Redis**, **Node.js** 등이 싱글 스레드임에도 엄청나게 빠른 비결.
* **비동기 처리**: I/O 대기 시간 동안 다른 로직을 처리하여 시스템 전체 효율을 극대화함.

---

### 3. 목적 (Why?)
| 구분 | 내용 |
| :--- | :--- |
| **자원 효율성** | 접속자마다 스레드를 만들지 않아 메모리 낭비가 없음. |
| **오버헤드 절감** | 스레드 간의 잦은 **컨텍스트 스위칭(Context Switching)** 비용을 최소화. |
| **응답성 향상** | 시스템이 I/O에 묶여(Blocking) 멈춰있는 시간을 없애 처리량을 높임. |

---

### 4. 구현 방식 (진화 과정)
1.  **select**: 전체 리스트를 매번 훑어봄 (비효율적이지만 표준).
2.  **poll**: 리스트 크기 제한은 없앴으나 여전히 순회 방식.
3.  **epoll / kqueue**: **이벤트가 발생한 놈들만** 따로 리스트를 만들어 알려줌 (가장 진화된 방식).

---

**💡 한 줄 결론** 단순히 기다리는 노가다는 **커널**에게 전담시키고, **CPU**는 커널이 '준비 완료' 신호를 보낼 때만 즉각 투입되어 데이터를 처리하는 효율적인 분업 시스템입니다.
